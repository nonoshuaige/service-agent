## 一、项目概述

### 1.1 项目背景

在现有校园活动票务系统（Spring Boot + Vue 3）基础上，基于 LangGraph 构建了一个 **Supervisor 多 Agent 系统**。系统自动识别用户意图，路由到售前/售后子 Agent，子 Agent 之间可通过 handoff 机制传递任务。

### 1.2 核心理念

- **Supervisor 路由**：rewrite_intent → supervisor → chitchat | pre_sales | after_sales
- **子 Agent 自治**：每个子 Agent 有自己的 system prompt + 工具集 + 内部 tool-calling loop
- **跨 Agent 协作**：子 Agent 通过 handoff 工具互相转接任务，supervisor 重新调度

### 1.3 核心功能

| 模块 | 功能 |
|------|------|
| 意图识别 | LLM 改写查询 + 分类意图（chitchat / pre_sales / after_sales） |
| Supervisor 路由 | LLM 调度器决定由哪个子 Agent 处理当前请求 |
| 售前 Agent | 搜索活动、购票下单、计算器 |
| 售后 Agent | 查询订单、票务政策FAQ、计算器 |
| 闲聊 Agent | 日常对话，无工具，友好回复 |
| Handoff | 子 Agent 相互感知，可传递任务（抑制回环：最多 1 次转接） |
| 三层记忆 | 压缩摘要 + 滑动窗口 recent + MySQL 全量持久化 |
| 性能优化 | 关键词快速通道（问候语跳过 LLM）+ 快慢模型分离（路由/闲聊用 flash 模型） |
| 前端集成 | SSE 流式输出（thinking/intent/route/tool_call/tool_result/final/done） |

### 4.2 Supervisor 路由流程

```
用户消息 → rewrite_intent_node (关键词快速通道 / LLM 改写 + 意图分类)
  → supervisor_node (chitchat 快速通道 / LLM 路由调度)
  → chitchat_agent (flash 模型, 1 轮) | pre_sales_agent | after_sales_agent
  → 子Agent内部 tool-calling loop (LLM ↔ 工具执行, max 5 轮, 完整模型)
  → 如有 handoff → supervisor 重新调度 (max 1 次)
  → SSE 流式推送 → save_turn (MySQL + Redis 双写)
```

### 4.3 工具分配

| Agent | 工具 |
|-------|------|
| chitchat | 无 |
| pre_sales (售前) | search_events, create_order, calculator, handoff_to_after_sales |
| after_sales (售后) | query_my_orders, calculator, handoff_to_pre_sales |

### 4.4 认证流程

1. 用户在票务系统登录 → Spring Boot 返回 JWT
2. 前端存储 token 到 `localStorage.authToken`
3. 请求 Agent API 时 Axios 拦截器附加 `Authorization: Bearer <token>`
4. Agent FastAPI `deps.py` 验证 JWT → 提取 `sub` 作为 userId → `ContextVar` 存储 token
5. Agent 工具调用 Spring Boot 后端时通过 `current_jwt.get()` 获取 token 转发

## 五、Agent 核心设计

### 5.1 意图识别 (intents.py)

`rewrite_intent_node` 作为 Graph 入口节点：
1. **关键词快速通道**：问候语（"你好"/"嗨"/"在吗"等）直接判定 chitchat，跳过 LLM；售前/售后关键词直接判定对应意图
2. 非明显意图 → LLM 调用（含对话历史最后 6 条用于代词消解），使用 flash 模型
3. 返回结构化结果：`{rewritten_query, intent, reason}`
4. 解析失败时 fallback 到关键词启发式分类

意图分类规则：chitchat（闲聊/问候/自我介绍）、pre_sales（查活动/买票）、after_sales（查订单/售后）

### 5.2 Supervisor 调度 (supervisor.py)

`supervisor_node` 决定下一步路由：
- **快速通道**：意图为 chitchat 且无 handoff → 直接路由 chitchat，跳过 LLM
- 正常情况：LLM 调度决策（使用 flash 模型）
- Handoff 请求：无条件路由到目标 Agent
- 完成判断：路由到 finish

### 5.3 子 Agent Loop (sub_agents/__init__.py)

所有子 Agent 共享 `_run_agent_loop()`：
1. 构建 LLM 输入（Agent 专属 system prompt + 对话历史 + handoff 上下文）
2. LLM 调用（绑定工具） → tool_calls 检测
3. 工具执行：普通工具 → 执行并回传；handoff 工具 → 设置 handoff_to 标记，退出 loop
4. 无 tool_calls → Agent 完成
5. Handoff 抑制：`handoff_count >= 1` 时禁止再次 handoff

### 5.4 System Prompt 设计

所有 prompt 集中在 `agent/prompts.py`，采用前缀缓存友好布局（静态模板在前，动态数据在后）：

- **REWRITE_INTENT_PROMPT**：改写规则 + 意图分类规则 + 边界判断
- **SUPERVISOR_PROMPT**：可用子Agent列表 + 路由决策规则
- **CHITCHAT_PROMPT**：主助手角色（非闲聊模块），能查活动、买票、查订单、闲聊。简洁温暖，禁止编造信息
- **PRE_SALES_PROMPT**：售前角色 + 工具使用规则 + 跨Agent协作（handoff_to_after_sales）
- **AFTER_SALES_PROMPT**：售后角色 + 工具使用规则 + 票务政策FAQ + 跨Agent协作

### 5.5 快慢模型分离 (v1.6)

为降低延迟，系统使用两级模型策略：
- **flash 模型** (`SUMMARY_MODEL`, glm-4.5-air)：意图识别、Supervisor 路由、闲聊 Agent
- **完整模型** (`ZHIPU_MODEL`, glm-4.5-air)：售前 Agent、售后 Agent（需工具调用和复杂推理）

`create_llm(fast=True)` 控制模型选择。目前两个模型相同，后续可替换为更快的专用模型。

### 5.6 Handoff 机制

1. 每个子 Agent 挂载对端 handoff 工具（如 pre_sales 挂 `handoff_to_after_sales`）
2. 子 Agent 内部 tool loop 检测 handoff → 设置 `state.handoff_to` → 退出 loop
3. Router 函数检查 `handoff_to` 非空 → 返回 `"supervisor"` 重新调度
4. Supervisor 处理 handoff → 清空 handoff 标记 → 路由到目标 Agent
5. `handoff_count` 计数器防止无限回环（最多 1 次转接）

## 六、双层存储与数据模型

v1.6 将 SQLite 替换为 MySQL，数据持久化能力与可靠性大幅提升。Redis 热数据层保持不变。

### 6.1 Redis Key 设计（全 Key 7 天 TTL）

| Key | 类型 | 内容 |
|-----|------|------|
| `agent:active_session:{userId}` | String | 当前活跃 session_id |
| `agent:recent:{userId}:{sessionId}` | List | 混合条目（消息 + 压缩摘要），最多 30 条 |
| `agent:sessions:{userId}` | ZSET | 最新 10 个 {sessionId: 更新时间} |
| `agent:session:{sessionId}` | Hash | agent_type / title / created_at 等元数据 |

### 6.2 MySQL 数据表（agent_db）

```sql
CREATE TABLE agent_messages (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id VARCHAR(64) NOT NULL,
    session_id VARCHAR(64) NOT NULL,
    role VARCHAR(16) NOT NULL,       -- 'user' | 'assistant'
    content TEXT NOT NULL,
    sequence_num INT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_msg_session (session_id, sequence_num)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE agent_compressions (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id VARCHAR(64) NOT NULL,
    session_id VARCHAR(64) NOT NULL,
    start_seq INT NOT NULL,
    end_seq INT NOT NULL,
    summary TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_comp_session (session_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 6.3 Redis 兜底恢复 (v1.6)

`load_session_messages` 如果发现 MySQL 中无消息（如旧会话仅存于 Redis），自动从 Redis 热数据恢复到 MySQL。确保历史会话始终可查。

### 6.4 滑动窗口 + 异步压缩

recent List 最多 30 条，满 20 条触发异步 LLM 压缩。压缩覆盖最旧的 20 条 → 1 条 compressed entry，写入 `agent_compressions` 表。

## 七、Agent API 设计

https://chatgpt.com/c/69bde1d2-7c7c-43f7-9ebe-59efc3f3100a

基础路径：`/api/v1/agent`  认证：`Authorization: Bearer <JWT>`

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /sessions | 获取会话列表（含 active_session_id） |
| POST | /sessions | 创建新会话 |
| PATCH | /sessions/{sessionId} | 重命名会话 |
| DELETE | /sessions/{sessionId} | 删除会话 |
| GET | /sessions/{sessionId} | 会话详情（最近 20 条 + has_more） |
| GET | /sessions/{sessionId}/messages?before_seq=N | 游标分页加载更早消息 |
| POST | /chat/stream | 发送消息，SSE 流式返回 |

### 流式对话（SSE）

```
POST /api/v1/agent/chat/stream
Content-Type: application/json

Request:
{
  "session_id": "sess_a1b2c3",  // null 则自动创建
  "message": "帮我查一下最近有什么活动？"
  // agent_type 已废弃，系统自动识别意图
}
```

### SSE 事件类型

| Event | 触发时机 | 前端行为 |
|-------|---------|---------|
| `thinking` (step=start) | 开始处理 | 显示 loading |
| `thinking` (step=intent) | 意图识别完成 | 显示识别结果 |
| `thinking` (step=route) | 路由确定 | 显示调度信息 |
| `tool_call` | 子Agent调用工具 | 显示工具名称和参数 |
| `tool_result` | 工具返回结果 | 显示结果（可折叠） |
| `final` | Agent输出最终答案 | 显示回复气泡 |
| `error` | 发生错误 | 显示错误提示 |
| `done` | 流结束 | 结束 loading |

## 附 B：常见问题

| 问题 | 解决 |
|------|------|
| conda 创建环境慢 | 配置国内镜像：`conda config --set show_channel_urls yes` |
| SSL 证书错误 | `SSL_CERT_FILE` 可能指向不存在的路径。运行 `python -c "import certifi; print(certifi.where())"` 获取正确路径并设置环境变量 |
| Redis 连接失败 | `docker ps` 确认 redis 容器在运行，`docker start redis` |
| JWT 验证失败 | 确认 `.env` 中 `JWT_SECRET` 与 Spring Boot 后端一致 |
| 前端 SSE 跨域 | FastAPI 配置 CORS 允许 `localhost:5173` |
| Agent 不调用工具 | LLM provider 兼容性问题，尝试切换 `LLM_PROVIDER=deepseek` |
| Handoff 死循环 | `handoff_count` 计数器限制最多 1 次转接，超出自动抑制 |
| MySQL 连接失败 | 确保 MySQL 服务已启动，`agent_db` 库已创建，`.env` 中连接参数正确 |
| query_my_orders 返回空 | Spring Boot 后端未运行或 `/api/v1/order/list` 接口不可用 |
