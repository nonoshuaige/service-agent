## 一、项目概述

### 1.1 项目背景

在现有校园活动票务系统（Spring Boot + Vue 3）基础上，新增一个基于 LangGraph 的智能 Agent 对话系统。Agent 能够调用工具执行任务（查询活动、下单、查订单等），具备三层记忆体系（长期记忆 + 会话摘要 + 近期明细），并通过 Redis Pipeline 实现高并发上下文读取。

### 1.2 核心理念

- **Agent 本质**：基于 LLM 的高级状态机 + Prompt 动态拼凑器
- **纵向隔离**：不同 Agent（如票务 Agent、客服 Agent）拥有各自独立的 toolList
- **横向共享**：所有 Agent 共享同一个用户的统一上下文中心，切换 Agent 时记忆不丢失

### 1.3 核心功能

| 模块 | 功能 |
|------|------|
| 对话管理 | 创建会话、浏览历史会话、删除会话 |
| Agent 对话 | ReAct 推理循环、工具调用、流式输出 |
| 思考过程 | 实时展示推理步骤，完成后折叠，点击可展开 |
| 三层记忆 | 长期记忆（向量召回）+ 会话摘要 + 近期明细滑动窗口 |
| 多 Agent | 票务助手（查活动/下单/查订单）、通用助手等，按需切换 |
| 前端集成 | 现有 Vue 前端新增 AI 助手入口，复用现有认证体系 |

### 4.2 与现有系统的集成点

| 集成点 | 方式 |
|--------|------|
| 用户认证 | Agent 后端验证 JWT Token（共享 Secret），从 token 提取 userId |
| Redis | 共用同一 Redis 实例，使用不同 key 前缀（`agent:` 前缀）避免冲突 |
| 工具调用 | Agent Tool 内部通过 HTTP 调用 Spring Boot 后端 API（如查询活动、下单） |
| 前端 | 现有 Vue 项目新增 `/agent` 路由，BottomNav 第三 Tab 为"助手"（位于笔记与我的之间） |

### 4.3 认证流程

1. 用户在票务系统登录 → Spring Boot 返回 JWT（Cookie + response body）
2. 前端存储 token 到 `localStorage.authToken`
3. 用户进入 AI 助手页面 → 前端 Axios 拦截器自动附加 `Authorization: Bearer <token>`
4. Agent FastAPI 后端通过 `deps.py` 依赖注入提取 JWT：
   - 解析 `Authorization: Bearer <token>` 头
   - `jwt_handler.verify_token()` 验证签名 + 算法 + `exp`/`iat` 时效
   - 提取 `sub` 字段作为 userId
   - 将原始 token 存入 `ContextVar("current_jwt")` 供下游工具使用
5. Agent 以 userId 为维度管理会话和记忆（Redis key 命名空间隔离）
6. Agent 工具调用 Spring Boot 后端时，通过 `current_jwt.get()` 获取 token 并附加 `Authorization` 头，实现全链路身份传递

## 五、Agent 核心设计

### 5.1 整体流程

```
用户消息 → FastAPI SSE 接口
  → 加载上下文（长期记忆 + 会话摘要 + 近期消息）
  → 构建 System Prompt（含工具使用规则 + 可用工具列表）
  → llm.ainvoke(messages)  → 一次性获取完整响应（可靠检测 tool_calls）
  → 如果有 tool_calls：
      → 执行工具（通过 ContextVar 获取 JWT 转发到 Spring Boot 后端）
      → 将 AI 响应 + ToolMessage 追加到消息历史
      → llm.ainvoke(messages) 获取最终回复
  → SSE 流式推送：thinking → tool_call → tool_result → final → done
  → 保存对话回合到 Redis
```

> **设计决策**：使用 `ainvoke`（非流式）进行首次 LLM 调用以可靠检测 tool_calls。流式 `astream` 的 chunk 在 OpenAI 兼容协议下，tool_calls 数据被拆分为 delta 增量，不同提供商（智谱/DeepSeek）行为不一致，导致 tool_calls 经常漏检。工具执行后，以非流式方式获取最终回复，SSE 事件粒度足以满足前端展示需求。

### 5.2 System Prompt 设计

系统提示词采用前缀缓存友好布局，静态模板在前，动态数据在后：

```python
def build_system_prompt(long_term, summary, tools_desc):
    parts = [
        "你是校园活动票务智能助手...",
        "",
        "## 重要：工具使用规则",
        "当用户询问以下内容时，你必须调用对应的工具获取真实数据：",
        "- 查询活动/演出/晚会/比赛 → 调用 search_events",
        "- 查询我的订单 → 调用 query_my_orders",
        "- 购买门票 → 调用 create_order",
        "- 数学计算 → 调用 calculator",
        "只要涉及以上场景，就先调用工具，再基于工具返回的真实结果回答。",
        "",
        "## search_events 使用技巧",
        "当用户想找「热卖」「热门」「推荐」的活动时，这些不是活动名称，",
        "直接用 keyword=\"\" 获取全部活动，然后从返回结果中向用户推荐。",
        "只有明确提到具体活动名称时，才用这些词做 keyword。",
    ]
    ...
```

> **关键改进**：v1.2 增加了明确的工具使用规则和 search_events 使用技巧，解决了 GLM-4-Flash 将"热卖"等抽象概念直接作为 keyword 传参的问题。

### 5.3 JWT 全链路传递

使用 Python `ContextVar` 实现 JWT 从 API 层到工具层的无侵入传播：

```python
# auth/jwt_handler.py
from contextvars import ContextVar
current_jwt: ContextVar[str] = ContextVar("current_jwt", default="")

# api/deps.py — 验证 JWT 后存储到 ContextVar
async def get_current_user_id(request: Request) -> str:
    ...
    current_jwt.set(token)
    return user_id

# tools/ticket_tools.py — 工具调用后端时读取并转发
def _auth_headers() -> dict:
    token = current_jwt.get()
    if token:
        return {"Authorization": f"Bearer {token}"}
    return {}
```

> 这解决了 v1.0 中 `query_my_orders` 和 `create_order` 必须保持为 stub 的核心问题——工具层无法获取用户 JWT 来调用需要身份认证的后端接口。

## 六、三层存储与数据模型

### 6.1 Redis Key 设计

所有 Agent 相关 Key 使用 `agent:` 前缀，与票务系统的 Key 隔离：

```
agent:long_term:{userId}           # Hash   长期记忆（偏好/背景）
agent:summary:{userId}:{sessionId} # String 会话摘要（异步生成）
agent:recent:{userId}:{sessionId}  # List   近期明细（滑动窗口，最多 50 条）
agent:sessions:{userId}            # ZSET   用户会话列表（score=更新时间）
agent:session:{sessionId}          # Hash   会话元数据（标题/创建时间/agent类型）
agent:active_session:{userId}      # String 用户上次停留的会话ID（用于恢复上下文）
agent:tools:{agentType}            # Hash   工具集缓存（减少重复序列化开销）
```

### 6.2 活跃会话追踪

用户上次停留在哪个会话，通过 `agent:active_session:{userId}` 记录。触发时机：

- **创建会话** → 自动将新会话设为活跃
- **查看会话详情** → 更新活跃指针
- **发送消息** → 更新活跃指针
- **删除会话** → 若删除的是当前活跃会话，清空指针

前端进入助手页面时，调用 `GET /sessions` 获取 `active_session_id`，若存在则自动加载该会话的消息历史，实现"回到上次离开的对话"。

### 6.3 各层详解

| 层级 | Key | 类型 | 刷新策略 | Prompt 作用 |
|------|-----|------|---------|-------------|
| 长期记忆 | `agent:long_term:{userId}` | Hash (field=类别, value=内容) | 用户上线时从向量库/DB 预加载；对话中根据 query 异步召回 Top-K | 决定语气、用户背景偏好 |
| 会话摘要 | `agent:summary:{userId}:{sessionId}` | String | 近期明细超过 10 轮时，异步调用轻量模型生成摘要并覆盖 | 携带长对话全局大纲，防止遗忘 |
| 近期明细 | `agent:recent:{userId}:{sessionId}` | List (LPUSH + LTRIM) | 每次对话追加，Redis 存 50 条，Prompt 取最近 6~10 条 | 保持当前对话精确语境 |

### 6.3 Redis Pipeline 优化

将 3 次独立 Redis 请求合并为 1 次 Pipeline：

```python
async def load_context(redis_client, user_id: str, session_id: str):
    """一次 Pipeline 获取所有上下文数据"""
    async with redis_client.pipeline() as pipe:
        pipe.hgetall(f"agent:long_term:{user_id}")
        pipe.get(f"agent:summary:{user_id}:{session_id}")
        pipe.lrange(f"agent:recent:{user_id}:{session_id}", 0, 9)
        results = await pipe.execute()
    
    return {
        "long_term": results[0],    # 长期记忆
        "summary": results[1],       # 会话摘要
        "recent": results[2],        # 最近 10 条明细
    }
```

> 与传统逐个查询相比，Pipeline 将网络往返从 3 次降为 1 次，内网 RT 控制在几 ms。

### 6.4 前缀缓存友好布局

智谱和 DeepSeek 的 API 均会在服务端对请求 prefix 进行自动 KV Cache 匹配。我们通过精心构造 Prompt 顺序来最大化缓存命中率：

```python
from langchain_core.messages import SystemMessage, HumanMessage, AIMessage

def build_messages(long_term: dict, summary: str, recent: list, tools_desc: str, user_query: str) -> list:
    """构造消息列表。静态系统提示词放最前面，利用服务端前缀缓存。"""
    system_prompt = f"""你是校园票务智能助手。

## 用户偏好
{format_long_term(long_term)}

## 对话摘要
{summary or "新对话"}

## 可用工具
{tools_desc}
"""
    return [
        # 第 1 条：系统提示词（高频静态内容 → 服务端自动缓存前缀 KV）
        SystemMessage(content=system_prompt),
        # 第 2~N 条：近期明细（变化频率高，不期望缓存命中）
        *[HumanMessage(content=m["query"]) if m["role"] == "user" else AIMessage(content=m["answer"])
          for m in recent],
        # 末条：当前用户输入
        HumanMessage(content=user_query),
    ]
```

> **缓存策略要点**：
> - **智谱 GLM-4**：服务端自动缓存相同 prefix 的 KV，无需 API 参数控制。将固定的系统提示词放在 `messages[0]` 即可命中。
> - **DeepSeek**：支持 Prompt Cache（context caching），首次请求完整计算，后续相同 prefix 请求自动复用，TTL 约 5~10 分钟。同样只需将系统提示词放在最前端。
> - **关键约定**：同一 Agent 类型的所有请求使用**完全相同的系统提示词模板**（仅替换动态占位符如 {long_term}、{summary}），确保 prefix 稳定以最大化缓存命中率。
> - **效果**：系统提示词 + 工具定义通常占 500~1500 Token，命中缓存后这些 Token 不再重复计算，首 Token 延迟可缩短 30%~50%。

event: thinking
data: {"step": 2, "content": "已获取活动列表，现在整理回复给用户"}

event: final

data: {}

### 7.3 SSE 事件类型说明

| Event | 触发时机 | 前端行为 |
|-------|---------|---------|
| `thinking` | Agent 开始每一步推理 | 显示思考块，标记"思考中"，实时追加文本 |
| `tool_call` | Agent 决定调用工具 | 在思考块中显示工具名称和参数 |
| `tool_result` | 工具执行完毕 | 在思考块中显示工具返回结果（可折叠） |
| `final` | Agent 输出最终答案 | 折叠/收起思考块，显示最终回复气泡，附带 session_id |
| `error` | 发生错误 | 显示错误提示 |
| `done` | 流结束 | 结束 loading 状态 |

### 8.5 前端 SSE 接收逻辑

```javascript
// api/agent.js
export async function sendMessageStream(sessionId, agentType, message, callbacks) {
  const token = localStorage.getItem('authToken');

  const response = await fetch('/api/v1/agent/chat/stream', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${token}`,
    },
    body: JSON.stringify({ session_id: sessionId, agent_type: agentType, message }),
  });

  const reader = response.body.getReader();
  const decoder = new TextDecoder();
  let buffer = '';

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    buffer += decoder.decode(value, { stream: true });
    const lines = buffer.split('\n');
    buffer = lines.pop(); // 保留未完成行

    for (const line of lines) {
      if (line.startsWith('event: ')) {
        currentEvent = line.slice(7).trim();
      } else if (line.startsWith('data: ')) {
        const data = JSON.parse(line.slice(6));
        // 根据 currentEvent 调用对应回调
        callbacks[currentEvent]?.(data);
      }
    }
  }
}
```

## 附 B：常见问题

| 问题 | 解决 |
|------|------|
| conda 创建环境慢 | 配置国内镜像：`conda config --set show_channel_urls yes` + 修改 `.condarc` |
| pip install 超时 | 设置代理：`set http_proxy=http://127.0.0.1:7897` |
| 智谱 API 调用失败 | 检查 `ZHIPU_API_KEY` 格式（`xxx.yyy` 点号分隔），确认 base_url 为 `https://open.bigmodel.cn/api/paas/v4/` |
| DeepSeek API 调用失败 | DeepSeek 服务端可能繁忙，在 `.env` 中切换 `LLM_PROVIDER=zhipu` 使用备用提供商 |
| API 返回 404 path not found | 智谱/DeepSeek 的 chat completions 路径不同：智谱需 `/chat/completions`（`langchain-openai` 自动拼接），检查 base_url 结尾是否有 `/v4/` |
| 流式输出无 data 返回 | 确保 `ChatOpenAI` 构造时设置 `streaming=True`，且 `stream=True` 参数传递正确 |
| Redis 连接失败 | `docker ps` 确认 redis 容器在运行，`docker start redis` |
| JWT 验证失败 | 确认 `.env` 中 `JWT_SECRET` 与 Spring Boot 后端一致 |
| 前端 SSE 跨域 | FastAPI 配置 CORS 允许 `localhost:5173` |
| 前缀缓存命中率低 | 同一 Agent 类型使用完全相同的系统提示词模板，避免在 prefix 区域拼接 session 级别动态内容 |
| Agent 不调用工具 | ① 检查 System Prompt 是否包含明确的工具使用规则；② 确认使用 `ainvoke` 而非 `astream`（流式 chunk 中 tool_calls 不可靠）；③ 检查 `.pyc` 缓存是否过期，清理 `__pycache__/` 后重启 |
| 工具调用报 NameError | 旧版本 `AIMessage` 未 import，`.pyc` 缓存残留。清理所有 `__pycache__/` 目录并重启服务 |
| LLM 把"热卖"当 keyword 搜索 | 在 System Prompt 和工具 description 中明确说明：抽象概念（热卖/热门/推荐）传空 keyword 获取全部活动，只有具体活动名才用作 keyword |
| 中文引号 `""` 导致 SyntaxError | Python 将全角引号 U+201C/U+201D 解析为字符串分隔符，改用 `「」` 或转义 |
| JWT 无法传递到工具层 | 使用 `contextvars.ContextVar` 在 deps.py 中 set，在工具中 get，无需侵入业务逻辑 |
| Token 永不过期 | `jwt.decode()` 默认不验证 exp。需显式设置 `options={"verify_exp": True, "verify_iat": True}` |
